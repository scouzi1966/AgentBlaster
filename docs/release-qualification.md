# Release Qualification Bundles

Release qualification bundles collect redaction-safe evidence for CI promotion, AFM release checks, media-supporting benchmark claims, or corporate review.

## Create A Bundle

```bash
agentblaster release qualification-bundle   --name afm-release   --evidence-bundle evidence/toolsim.agentblaster-evidence.zip   --provider-contract-matrix reports/qwen-gemma-provider-contract-matrix.json   --comparison-gate reports/comparison-gate.json   --matrix-gate reports/qwen-gemma-matrix-gate.json   --telemetry-audit reports/run-telemetry-audit.json   --normalized-telemetry reports/afm-normalized-telemetry.json   --matrix-pressure-audit reports/qwen-gemma-stress-pressure.json   --matrix-saturation-report reports/qwen-gemma-matrix-saturation.json   --matrix-scorecard reports/qwen-gemma-matrix-scorecard.json   --implementation-status reports/implementation-status.json   --campaign-preflight-manifest campaign-preflight/qwen-gemma-local/manifest.json   --benchmark-readiness reports/afm-trace-readiness.json   --engine-advisory reports/afm-improvement-plan.json   --evidence-index reports/afm-release-evidence-index.json   --suite-audit reports/toolsim-suite-audit.json   --metric-coverage reports/afm-metric-coverage.json   --release-provenance reports/release-provenance.json   --publication-bundle publication-bundles/run.agentblaster-publication.zip   --matrix-publication-bundle publication-bundles/qwen-gemma-matrix-summary.agentblaster-matrix-publication.zip   --harness-review reports/harness-contract-fuzz-review.json   --selftest-report test-reports/selftest/selftest-report.json   --sdlc-validation-manifest test-reports/sdlc-validation-manifest.json   --output-dir release-bundles   --audit-log audit/control-plane.jsonl
```

## Allowed Artifact Categories

- Evidence bundles ending in `.agentblaster-evidence.zip`.
- Publication bundles ending in `.agentblaster-publication.zip`.
- Publication brief JSON artifacts.
- Matrix publication bundles ending in `.agentblaster-matrix-publication.zip` with `matrix-publication-bundle-manifest.json`.
- Executed provider contract-check or provider contract-matrix JSON artifacts.
- Comparison gate JSON artifacts.
- Matrix gate JSON artifacts.
- Telemetry audit JSON artifacts.
- Matrix pressure audit JSON artifacts.
- Matrix saturation report JSON artifacts.
- Engine improvement advisory JSON artifacts.
- Evidence index JSON artifacts.
- Suite audit JSON artifacts.
- Metric coverage JSON artifacts.
- Claim readiness JSON artifacts.
- Release provenance JSON artifacts.
- Selftest report artifacts.
- SDLC validation manifest JSON artifacts.

The generated `manifest.json` records every artifact category, redacted source filename, archive path, SHA-256 checksum, byte size, schema, status, and status source. It also records top-level `ok`, aggregate `artifact_status`, and blocking status names. JSON artifact contents are not embedded in the manifest. Matrix gate artifacts must use `agentblaster.matrix-gate.v1`; unversioned or stale matrix-gate files are marked `invalid-schema`, which blocks bundle `ok`. Matrix gate artifacts may contribute a compact `review_summary` containing only `schema_version`, `failure_class_summary`, `failure_class_artifacts_missing`, `tool_loop_stop_summary`, `tool_loop_artifacts_missing`, judge-rubric verdict counts, tool-parser-repair counts, class-specific class/metric/actual/threshold gate findings, matrix name, and pass rate so reviewers can understand release blockers without opening normalized result artifacts. Matrix pressure artifacts must use `agentblaster.matrix-pressure-audit.v1` and may contribute only compact aggregate prompt-pressure metadata: run/case counts, scheduled and concurrent-window prompt-token pressure, concurrency-weighted pressure, shared static prefix groups/tokens, `shared_static_reuse_tokens`, engines/models/suites/concurrency levels, and highest-pressure run summaries. Matrix saturation artifacts must use `agentblaster.matrix-saturation.v1` and may contribute only compact concurrency evidence: result artifact coverage, concurrency levels, max queue/rate-limit wait, finding counts, top pressure entries, and guidance. Matrix scorecard artifacts use `agentblaster-matrix-scorecard-v1` and may contribute only compact scorecard evidence: pass rate, result-artifact coverage, compact engine-target IDs, architecture and quantization rollups, judge-rubric counts, tool-parser-repair counts, failure/tool-loop counts, telemetry quality/readiness summary, and scorecard concurrency evidence. Normalized telemetry sample artifacts use `agentblaster.normalized-telemetry.v1`; release bundles archive a compact redacted copy containing review summary counts and stats-labeling posture only, not raw usage maps, raw stats maps, or source maps. Implementation-status artifacts are redacted at bundle time so `project_root` is replaced with `<redacted>` before the archived checksum is computed. Selftest artifacts use `agentblaster.selftest-report.v1` and may contribute only compact app-SDLC evidence: run id, tier, status, exit code, duration, browser metadata, marker expression, and JUnit presence. Publication briefs and SDLC validation manifests are archived as compact redacted copies; release manifests preserve only publication-brief readiness/count metadata, compact engine-target IDs, architecture/quantization rollups, protocol-repair posture, Chrome/GUI coverage counts, expected evidence counts, and security posture booleans. Full prompts, largest-case lists, scorecard leaderboard entries, selftest command strings, selftest environment maps, publication-brief proof text, publication-brief recommended wording, SDLC fixture commands, local source paths, raw result rows, raw telemetry usage/stats maps, telemetry source maps, and provider payloads are not copied into review summaries. Cleanup artifacts use `agentblaster.cleanup-plan.v1` or `agentblaster.retention-cleanup.v1` and should be routed through evidence indexes; the index preserves only compact execution mode, action count, selected-selector count, action types, and redaction-safe security booleans, not cleanup path lists. Run publication bundles are summarized from their embedded `publication-bundle-manifest.json` only; the release manifest records compact run ID, artifact names, media-kit asset roles, missing recommended assets, publication readiness status/counts, and security booleans for raw secrets, raw provider payloads, and `results.jsonl`. Matrix publication bundles are summarized from their embedded `matrix-publication-bundle-manifest.json` only; the release manifest records compact engine-target IDs, architecture/quantization rollups, matrix artifact names, media-kit asset roles, missing recommended assets, and security booleans for raw secrets, provider payloads, `results.jsonl`, and per-run raw traces. Publication readiness `blocked`, unsafe security flags, malformed publication manifests, invalid zip files, and oversized publication manifests block release-bundle `ok`; `review-required` remains a review status. Unsafe matrix publication security flags, malformed matrix publication manifests, invalid zip files, and oversized matrix publication manifests also block release-bundle `ok`; missing recommended run or matrix media-kit assets remain review status. Harness review artifacts may contribute only compact suite, generator, calibration, surface-count, and assertion-count metadata. Engine improvement advisories may contribute only compact engine, priority-count, highest-priority, no-dispatch, priority-area, and aligned-artifact metadata. Evidence indexes may contribute only compact packet name, artifact count, status counts, and readiness state/count metadata. Suite audits may contribute only compact suite, case count, provenance/risk counts, finding codes, finding count, and duplicate-fingerprint count. Free-form finding messages, advisory reason text, prompt text, and raw evidence rows are not copied. Other zip artifacts are classified by filename unless they are release qualification, run publication, or matrix publication bundles with capped embedded manifests.

## Packaging Readiness

Benchmark readiness dossiers use `agentblaster.benchmark-readiness.v1` and may be included with `--benchmark-readiness`. Release manifests preserve only compact readiness fields: provider, suite, model, ready state, strict-unknown state, policy/suite compatibility, contract-check planning counts, metric coverage score, blocking/warning counts, redacted provider-auth posture, and static secret-backend posture such as optional keyring dependency availability. Raw provider configs, secret names beyond backend kind, key values, headers, traces, prompts, and endpoint payloads are not copied into the manifest review summary.

`agentblaster release provenance` includes a static `packaging_readiness` section and an SPDX-lite `sbom` section. Packaging readiness checks package metadata, license/classifier metadata, Hatchling build configuration, the `agentblaster` CLI entrypoint, optional dependency groups for development, Parquet/result exports, GUI testing, PNG/media report rendering, and keyring support, pytest SDLC markers, and expected enterprise/release docs. The SBOM inventory is derived from `pyproject.toml` plus optional installed-package metadata and does not contact package indexes, read environment variables, copy provider configs, or include secret values.

Generate the same readiness report directly when CI needs a package metadata gate before building distributions:

```bash
agentblaster release packaging-readiness \
  --output-json reports/packaging-readiness.json \
  --fail-on-gaps \
  --audit-log audit/control-plane.jsonl
```

Capture implementation inventory before release evidence assembly:

```bash
agentblaster implementation-status --output-json reports/implementation-status.json
agentblaster catalog artifact-schemas --format json --output reports/artifact-schemas.json
```

This readiness report is static. It does not build wheels, install the package, import optional dependencies, contact package indexes, or run tests. Treat it as preflight evidence, then run the packaging selftest tier before publishing:

```bash
agentblaster selftest --tier packaging --report-dir test-reports/selftest/packaging
```

The repository package workflow (`.github/workflows/publish.yml`) runs the same static readiness gates before building source and wheel distributions. It uploads package artifacts and release reports to GitHub Actions artifacts only; it does not publish to PyPI.

## Safety Rules

Release qualification bundles reject obvious raw run artifacts such as `results.jsonl`, paths containing a `raw` segment, and unrecognized zip files. They are intended for review artifacts, not replay/debug bundles.
Provider contract checks, provider audits, implementation-status inventories, campaign-preflight manifests, benchmark readiness dossiers, telemetry audits, normalized telemetry samples, matrix pressure audits, matrix saturation reports, matrix scorecards, matrix gates, selftest reports, SDLC validation manifests, publication briefs, harness reviews, suite calibration reports, engine advisories, evidence indexes, suite audits, metric coverage reports, and claim-readiness reports are accepted as JSON review artifacts because they are derived from compact normalized statuses, normalized results, static matrix/suite definitions, calibration manifests, static implementation inventories, campaign planning summaries, app-SDLC execution metadata, or redacted provider posture and do not include raw traces or provider payloads. Matrix saturation reports may include `concurrency_evidence` with multi-level coverage, queue/rate-limit pressure, and top pressure entries derived from normalized result rows. Matrix scorecards may include parser-repair aggregate fields, `telemetry_quality_summary`, `stats_comparability_summary`, and `concurrency_evidence` so release reviewers can see protocol-repair behavior, comparability, stats-labeling, and saturation evidence without opening leaderboard entries or raw rows. Normalized telemetry samples are archived as compact redacted copies and release manifests preserve only contract, adapter, stats-profile, quality-count, comparison-guidance, and stats-labeling summary fields. Provider audits are archived as compact redacted copies and release manifests preserve only provider counts, remote counts, error/warning counts, finding codes, policy-control booleans, and auth-backend posture by backend kind; finding messages, raw secret-reference names, dotenv paths, environment values, and keyring contents are not copied. Implementation-status summaries include area counts, built-in harness-engineering and tool-parser-repair case counts, stats-comparability profile counts, and SDLC/security posture without copying project roots or file-evidence paths. Campaign-preflight manifests are archived as redacted compact copies and release manifests preserve only `manifest.review_summary` fields such as matrix/run/case counts, benchmark-readiness count, and publication-safety booleans; local output paths, matrix paths, policy paths, and dry-run commands are not copied. Benchmark readiness summaries may include redacted provider-auth posture by backend kind only, including writable backend and plaintext dotenv fallback counts. Metric-coverage summaries preserve compact comparability counts plus `claim_contract` leaderboard/disclosure posture, not per-field notes. Selftest reports may include command and environment metadata in the artifact, but release summaries do not copy those fields. Publication briefs and SDLC validation manifests are stored as compact release copies; publication-brief summaries may preserve compact engine-target IDs plus architecture/quantization rollups and protocol-repair posture, but proof text, recommended wording, fixture commands, raw test logs, and local paths are omitted. Run publication bundles are accepted only when their embedded publication manifest can be parsed and does not report unsafe contents. Suite calibration reports are accepted only as compact pass/finding summaries, and failed supplied calibration reports block release qualification. Release qualification manifests copy only compact review summaries, not `results.jsonl`, raw traces, prompts, responses, provider configs, API keys, request headers, implementation file evidence paths, campaign matrix paths, campaign dry-run commands, raw telemetry usage/stats maps, telemetry source maps, per-field metric notes, full scorecard entries, selftest command strings, selftest environment maps, publication-brief proof text, publication-brief recommended wording, SDLC fixture commands, raw test logs, or full matrix-pressure case lists.

## Final Redaction Gate

Run `agentblaster security scan release-bundles/<name>.agentblaster-release-qualification.zip --output-json reports/redaction-scan.json` before publishing or attaching a bundle externally. The scanner writes `agentblaster.redaction-scan.v1`, reports pattern names and locations without printing matched secret values, and checks ZIP text entries plus unsafe/sensitive ZIP member names without extracting archives. Claim readiness requires this schema-versioned scan artifact and carries only compact scan counts, skipped counts, finding counts, and pattern-count summaries into review output.

## Claim Readiness Gate

Before publishing a corporate or media-facing benchmark claim, collect the final evidence set into a claim-readiness report:

```bash
agentblaster release claim-readiness \
  --name qwen-gemma-local \
  --experiment-manifest reports/qwen-gemma-experiment.json \
  --experiment-gate reports/qwen-gemma-experiment-gate.json \
  --provider-contract-matrix reports/qwen-gemma-provider-contract-matrix.json \
  --matrix-gate reports/qwen-gemma-matrix-gate.json \
  --telemetry-audit reports/run-telemetry-audit.json \
  --normalized-telemetry reports/afm-normalized-telemetry.json \
  --matrix-pressure-audit reports/qwen-gemma-stress-pressure.json \
  --matrix-saturation-report reports/qwen-gemma-matrix-saturation.json \
  --matrix-scorecard reports/qwen-gemma-matrix-scorecard.json \
  --implementation-status reports/implementation-status.json \
  --benchmark-readiness reports/afm-trace-readiness.json \
  --release-provenance reports/release-provenance.json \
  --release-qualification-bundle release-bundles/qwen-gemma-local.agentblaster-release-qualification.zip \
  --redaction-scan reports/redaction-scan.json \
  --publication-bundle publication-bundles/run.agentblaster-publication.zip \
  --matrix-publication-bundle publication-bundles/qwen-gemma-matrix-summary.agentblaster-matrix-publication.zip \
  --harness-review reports/harness-contract-fuzz-review.json \
  --suite-calibration-report reports/harness-contract-fuzz-calibration-report.json \
  --engine-advisory reports/afm-improvement-plan.json \
  --evidence-index reports/afm-release-evidence-index.json \
  --suite-audit reports/toolsim-suite-audit.json \
  --metric-coverage reports/afm-metric-coverage.json \
  --campaign-preflight-manifest campaign-preflight/qwen-gemma-local/manifest.json \
  --selftest-report test-reports/selftest/selftest-report.json \
  --output-json reports/qwen-gemma-claim-readiness.json \
  --fail-on-blockers
```

After claim readiness is generated, create the executive/media/corporate brief from compact review artifacts:

```bash
agentblaster release publication-brief \
  --name qwen-gemma-local \
  --claim-readiness reports/qwen-gemma-claim-readiness.json \
  --matrix-scorecard reports/qwen-gemma-local-summary-matrix-scorecard.json \
  --release-provenance reports/release-provenance.json \
  --evidence-index reports/qwen-gemma-evidence-index.json \
  --output-json reports/qwen-gemma-publication-brief.json \
  --output-md reports/qwen-gemma-publication-brief.md
```

Publication briefs use `agentblaster.publication-brief.v1`. They summarize readiness status, scorecards, compact engine-target IDs, architecture/quantization rollups, protocol-repair posture, proof points, disclosures, recommended wording, and a security boundary without opening raw result rows, traces, provider configs, keyrings, dotenv files, or remote endpoints. Blockers and warnings are preserved as disclosures rather than hidden in polished output.

The gate checks that required planning, executed provider contract checks, runtime gate, telemetry comparability, matrix pressure, executed saturation, provenance, redaction, and release-bundle artifacts are present and passing. Provider contract evidence can be individual `agentblaster.provider-contract-check.v1` artifacts or one `agentblaster.provider-contract-matrix.v1` artifact, but supplied artifacts must report `ok: true`; plan-only contract-check artifacts are useful preflight material but block publication readiness. Claim readiness and release qualification preserve compact provider-contract capability evidence: directly checked capabilities, proxy coverage such as `judge_rubric` through `structured_output`, and capabilities still requiring separate benchmark evidence such as Anthropic `prompt_caching` and local-agent `tool_parser_repair`. Matrix gates must use `agentblaster.matrix-gate.v1` and report `ok: true`; claim-readiness output preserves their aggregate `failure_class_summary` counts, `failure_class_artifacts_missing`, `tool_loop_stop_summary` counts, `tool_loop_artifacts_missing`, judge-rubric verdict-validity counts, judge-rubric result-artifact gaps, and class-specific gate findings so reviewers can see whether accepted residual failures were model-quality issues, engine protocol bugs, feature gaps, runtime failures, harness defects, incomplete result-artifact retention, bounded tool-loop stop reasons, or evaluator-style rubric failures. Telemetry audits preserve run-level `comparison_readiness` so release claims can disclose native/measured fields separately from inferred, conditional, unknown, incomplete, or unavailable metrics. Normalized telemetry samples can be supplied directly with `--normalized-telemetry` or inherited from release qualification bundles; claim readiness copies only compact stats-comparability counts, quality counts, comparison guidance, and stats-labeling posture, and emits warnings when advisory, raw-provenance, or missing stats fields require disclosure. Matrix saturation evidence must use `agentblaster.matrix-saturation.v1` and report `ok: true`; warnings can remain in the artifact for advisory review, but error findings block publication readiness. Matrix scorecard evidence can be supplied directly with `--matrix-scorecard` or embedded in the release qualification manifest; its compact summaries preserve engine-target IDs, and missing scorecard evidence, advisory telemetry-quality labels, unknown telemetry quality, missing tool-parser repair coverage, invalid tool-call emissions, incomplete parser-repair validity, or single-level concurrency evidence create claim-readiness warnings rather than blockers. Implementation status artifacts must use `agentblaster.implementation-status.v1`; supplied artifacts and release-bundle embedded summaries block claim readiness when `missing_areas` is non-zero or `status` is not `implementation-ready-for-validation`, while passing artifacts are summarized without project roots or file-evidence paths. Campaign preflight manifests must use `agentblaster.campaign-preflight-bundle.v1`; supplied manifests and release-bundle embedded summaries are summarized without local matrix paths or dry-run commands and are used as static campaign planning evidence. The human-readable claim-readiness output includes `campaign_preflight:`, `matrix_publication_bundles:`, and `publication_briefs:` lines with compact matrix/run/case counts, media-kit posture, publication readiness, and engine-target IDs when available. Benchmark readiness dossiers must use `agentblaster.benchmark-readiness.v1`; supplied dossiers and release-bundle embedded readiness summaries block claim readiness when `ready` is not true, while passing dossiers are summarized by provider, suite, model, policy/suite compatibility, contract planning counts, metric coverage score, blocking/warning counts, and redacted provider-auth posture. Selftest reports must use `agentblaster.selftest-report.v1`; failed supplied selftests block claim readiness, while passing selftests are summarized by tier, run id, exit code, duration, browser metadata, and JUnit presence only. Supplied run publication bundles must end in `.agentblaster-publication.zip`; claim readiness reads only their capped `publication-bundle-manifest.json`, requires schema `agentblaster.publication-bundle.v1`, blocks manifests that report raw secrets, raw provider payloads, or `results.jsonl`, blocks `publication_readiness.status=blocked`, and surfaces `review-required` or missing recommended media-kit assets as claim-readiness warnings. Supplied matrix publication bundles must end in `.agentblaster-matrix-publication.zip`; claim readiness reads only their capped `matrix-publication-bundle-manifest.json`, requires schema `agentblaster.matrix-publication-bundle.v1`, blocks manifests that report raw secrets, raw provider payloads, `results.jsonl`, or per-run raw traces, and surfaces compact engine-target IDs plus missing recommended media-kit assets as warnings. Claim readiness also surfaces compact publication-bundle, matrix-publication-bundle, publication-brief, implementation-status, campaign-preflight, benchmark-readiness, normalized-telemetry, SDLC-validation, and selftest summaries already embedded in the release qualification `manifest.json`, and it can read compact benchmark-readiness summaries from a supplied campaign-preflight manifest's `benchmark_readiness.artifact_path`. Harness review, suite-calibration, engine advisory, evidence-index, suite-audit, metric-coverage, and normalized-telemetry artifacts are optional, but supplied artifacts must use `agentblaster.harness-review.v1`, `agentblaster.suite-calibration-report.v1`, `agentblaster.engine-improvement-advisory.v1`, `agentblaster.evidence-index.v1`, `agentblaster.suite-audit.v1`, `agentblaster.metric-coverage.v1`, and `agentblaster.normalized-telemetry.v1` so generated-suite calibration, release-gate approval, engine-roadmap priorities, packet-level evidence status, cleanup evidence rollups, dataset-hygiene findings, metric comparability disclosure, and stats-comparability sample evidence are reviewable. A supplied evidence index with readiness state `blocked` blocks claim readiness; `review-required` is allowed but remains visible for human signoff. A supplied suite calibration report that did not pass blocks claim readiness when it is used as release-gate evidence. A supplied suite audit with governance findings adds claim-readiness warnings but does not hide those findings inside model or engine scores. A supplied metric coverage report with review-required comparability groups adds claim-readiness warnings so publication claims disclose inferred, conditional, partial, or unavailable metric families. Claim readiness copies only compact summary metadata, not advisory reason text, suite-audit finding messages, implementation file evidence paths, campaign matrix paths, campaign dry-run commands, prompt text, raw evidence payloads, full scorecard leaderboard entries, selftest command strings, selftest environment maps, publication-brief proof text, SDLC fixture commands, raw provider configs, API-key values, full secret reference names, raw telemetry usage/stats maps, telemetry source maps, raw test logs, or per-field metric notes. Absolute and parent-relative input paths are reduced to safe artifact names with `path_redacted=true` in checks. Claim readiness opens only `manifest.json` inside the release qualification zip, `publication-bundle-manifest.json` inside run publication bundles, and `matrix-publication-bundle-manifest.json` inside matrix publication bundles, caps their sizes, and does not extract bundled evidence files. It reads only supplied review artifacts; it does not inspect raw traces, provider configs, keyrings, or remote endpoints.

For large generated campaigns, `--benchmark-readiness-list campaigns/<name>/reports/benchmark-readiness-inputs.txt` can replace many repeated `--benchmark-readiness` flags in both `release qualification-bundle` and `release claim-readiness`.

Claim-readiness reports are usually generated after the first release qualification bundle and redaction scan because they reference those artifacts. Publication briefs are usually generated after claim readiness. If you need a single archival zip that also contains `claim-readiness.json` and the final publication brief, create a second qualification bundle with `--claim-readiness reports/<claim>-claim-readiness.json`, `--publication-brief reports/<claim>-publication-brief.json`, and the relevant `--sdlc-validation-manifest` artifact.


## Experiment Context

For corporate or media-facing benchmark campaigns, create and gate an experiment manifest before execution:

```bash
agentblaster experiment manifest --name qwen-gemma-local --objective "Compare local agentic engines." --providers afm,lm-studio --targets qwen3.6-27b-dense,gemma-4-31b-dense --suites trace-replay,agentic-tool-loop,agent-fanout,prefill,cache-control,cancellation,lcp-context --policy agentblaster.policy.yaml --output reports/qwen-gemma-experiment.json
agentblaster experiment gate reports/qwen-gemma-experiment.json --require-policy --output-json reports/qwen-gemma-experiment-gate.json
```

Attach the manifest and gate report alongside evidence bundles and release qualification artifacts so reviewers can see the benchmark scope and acceptance gates that existed before execution.
